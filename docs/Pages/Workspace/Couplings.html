Couplings


A coupling is a link from an object in an attribute of an object in another component (or the same component). A producer produces values that are read by a consumer.

A list of all current couplings is shown in the Coupligns tab of the bottom information pane.

Most windows in Simbrain and workspace components, which are wrappers around other objects, which are designed to exist in a workspace.  This mainly means attributes (for couplings). Couplings tie these objects together via attributes. Attributes are the "hooks" or "anchor points" that make up couplings. There are two kinds, producing and consuming, which spit out and gobble up numbers (currently; may be abstracted to more general datatypes).  So, the workspace can be thought of as a bunch of things that spit out numbers and read them in. Making a simulation amounts to making number spitter-outers and number-readers.  For example, a simulation might involve coupling neurons whose neurons are producers to a bar chart whose bars are consumers (to see this, try bar chart test in the script menu).


Attributes

An attribute is generic term for a producer or consumer, something that produces values or consumes values.

Examples are neuron activations, weight strengths, or bars on a bar chart, or sensor values for an agent in a virtual world.

They are the parts of a coupling, attachments points in the simulation environment that can be connected together in the broader simulation.


Coupling Creation

There are a few different ways to create couplings.

1. Use the coupling manager. To open this press the coupling manager button on the Simbrain toolbar. This allows you to view every coupling in the workspace. Here is a screenshot of the current coupling manager. The current design is based on a mode where each component has its own list of “bound couplings.”  That will change, however.  (We should discuss this …)

2.  Via context menus inside specific components.  E.g. if you right click on a neuron in a neural network a context menu will point you to all other potential consuming and producing attributes in the whole workspace.  These context menus are built at the workspace menu by polling all current workspacecomponents.

3. Via "coupling" commands in some menus.  Quick and easy!  

4. Using a script.

Confusions

Sometimes the terminology can be confusing. For example, in order world, sensors seem to consume (info) and motors seem to produce (motion).   But in the context of the workspace it’s the opposite.   consumed info is produced or sent elsewhere... produced motion is based on consumed info from elsewhere.]
